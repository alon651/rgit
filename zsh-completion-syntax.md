# Zsh Completion Syntax Explained (for Beginners)

This guide explains every piece of syntax in a typical autogenerated Zsh completion file (like `_rgit`), assuming you know how to write a basic shell script but have never written a completion function before.

---

## 1. File Header: `#compdef rgit`

```zsh
#compdef rgit
```

This is a **magic comment**. When Zsh sees this on the first line of a file, it knows:
- This file provides completions for the command `rgit`.
- It will be automatically loaded when you type `rgit <TAB>`.

If you source this file manually (e.g. in `.zshrc`), Zsh registers the completions.

---

## 2. The Main Function: `_rgit()`

```zsh
_rgit() {
```

Zsh's completion system has a naming convention: **the completion function must start with an underscore**, followed by the command name. So for `rgit`, the function is `_rgit`.

When you press `<TAB>` after typing `rgit`, Zsh calls this function. Your job inside the function is to tell Zsh what completions to offer.

---

## 3. `_arguments`: The Core Function

`_arguments` is the workhorse of Zsh completions. You feed it a list of **spec strings** that describe valid arguments (flags, options, positional arguments).

### Basic call pattern

```zsh
_arguments [options] [separator] spec1 spec2 spec3 ...
```

Example from the file:
```zsh
_arguments "${_arguments_options[@]}" : \
    '-h[Print help]' \
    '--help[Print help]' \
    ':path -- The hash of the object:_default'
```

- `"${_arguments_options[@]}"` — options that control behavior (explained later).
- `:` — the separator. It marks the boundary between "normal options/arguments" and "rest arguments". Most of the time you don't need to worry about what goes after it; just know it's there.
- Everything after `:` is a spec string, continued across lines with `\`.

---

## 4. Spec Strings: The Building Blocks

A spec string is a single quoted string that describes one thing the user can type. There are several formats:

### 4.1 Flags (options that take NO value)

```zsh
'-h[Print help]'
'--help[Print help]'
```

Format: `'-<short_flag>[<description>]'` or `'--<long_flag>[<description>]'`

- The `-` or `--` prefix tells Zsh it's a flag.
- Text inside `[...]` is the description shown in the completion menu.
- The short flag `-h` and long flag `--help` are independent — Zsh offers both.

### 4.2 Flags that TAKE a value

```zsh
'-p+[The id of a parent commit object]:PARENT:_default'
'-m+[A paragraph in the commit log message]:MESSAGE:_default'
```

Format: `'<flag>+[<desc>]:<placeholder>:<action>'`

- The `+` after the flag name means **"this flag requires a value after it"**.
- After `[description]`, a colon `:` starts the **argument part**.
- `PARENT` / `MESSAGE` — placeholder text shown in the completion menu (like "what should the user type?").
- `_default` — the **action**. This is a function Zsh calls to generate completions for that value. `_default` is a built-in that does generic completion (usually filenames).

### 4.3 Positional Arguments

```zsh
':hash -- The hash of the object to read:_default'
```

Format: `':<label>:<action>'`

- The leading colon (with nothing before it, no `-` prefix) means this is a **positional argument** (not a flag).
- `hash -- The hash of the object to read` — the label/description shown to the user.
- `_default` — the action for generating completions.

### 4.4 Optional Positional Arguments

```zsh
'::path -- Data dir location:_files'
```

Format: `'::<label>:<action>'`

- Two colons `::` at the start mean this argument is **optional**.
- `_files` is a built-in action that completes file/directory paths.

### 4.5 Repeatable (Greedy) Arguments

```zsh
'*::paths -- files to add content from:_files'
```

Format: `'*::<label>:<action>'`

- The leading `*` means **"the user can type this multiple times"** (greedy).
- Example: `rgit add file1.txt file2.txt file3.txt`

---

## 5. Delegating to Another Function (Actions)

```zsh
":: :_rgit_commands"
```

Format: `'<spec-part>::<action-function>'`

- `_rgit_commands` is a **function name** (bare name, no quotes). When Zsh needs completions for this spec, it calls that function.
- `_files`, `_default`, etc. are built-in action functions. But you can write your own — like `_rgit_commands`.

### How `_rgit_commands` works

```zsh
_rgit_commands() {
    local commands; commands=(
        'init:Initialize the repo'
        'add:Add file content to the index'
        'commit:Commit changes to the repository'
        'help:Print this message or the help of the given subcommand(s)'
    )
    _describe -t commands 'rgit commands' commands "$@"
}
```

- `commands=( 'value:description' 'value:description' ... )` — an array of completion entries.
- `_describe` — Zsh built-in that takes an array and presents it as a completion menu. It shows the `description` next to each `value`.
- `-t commands` — a **tag** (grouping name for display/filtering).
- `'rgit commands'` — the group heading shown in the menu.
- `commands` — the array variable name.
- `"$@"` — forwards any extra arguments.

### Idempotent definition

```zsh
(( $+functions[_rgit_commands] )) ||
_rgit_commands() { ... }
```

This pattern means: "if the function `_rgit_commands` is NOT already defined, then define it."

- `$+functions[name]` returns 1 if the function exists, 0 if not.
- `(( ... ))` is arithmetic evaluation; 0 fails, non-zero succeeds.
- `||` runs the right side only if the left side fails (i.e., function doesn't exist).

This prevents redefinition conflicts if the file gets sourced multiple times.

## 6. The State Machine: Handling Subcommands

This is the trickiest part. `_arguments` can't describe "if first arg is X, then valid args are Y" natively. Instead, Zsh uses a **state machine** pattern.

### Step 1: Set up state dispatch in the spec

```zsh
_arguments ... : \
    '-h[Print help]' \
    '--help[Print help]' \
    ":: :_rgit_commands" \
    "*::: :->rgit" \
    ...
```

The key spec is:
```zsh
"*::: :->rgit"
```

Breaking this down:
- `*` — greedy (matches remaining args)
- `:::` — three colons
- `: ` — the label is empty
- `:->rgit` — this is the **action**. `->rgit` means "set the variable `$state` to `rgit` and return control to me."

### Step 2: Handle the state

After `_arguments` returns, `$state` is set to `rgit`. Now you write a `case` block:

```zsh
case $state in
(rgit)
    words=($line[1] "${words[@]}")
    (( CURRENT += 1 ))
    curcontext="${curcontext%:*:*}:rgit-command-$line[1]:"
    case $line[1] in
        (add)
            _arguments "${_arguments_options[@]}" : \
                '-h[Print help]' \
                '--help[Print help]' \
                '*::paths -- files to add content from:_files' \
            && ret=0
            ;;
        (commit)
            _arguments "${_arguments_options[@]}" : \
                '-m+[]:MESSAGE:_default' \
                '-h[Print help]' \
                '--help[Print help]' \
            && ret=0
            ;;
        ...
    esac
    ;;
esac
```

Let's unpack this:

- `$state` is set to `rgit` (from the `->rgit` action).
- `$line[1]` — the first subcommand the user typed (e.g., `add`, `commit`, `help`).
- `words=($line[1] "${words[@]}")` — reorders the word list to put the subcommand name first.
- `(( CURRENT += 1 ))` — adjusts the cursor position index because we shifted the words.
- `curcontext=...` — updates the completion context string (Zsh uses this internally for caching/grouping).
- Then a nested `case` dispatches based on which subcommand was typed, calling `_arguments` again with only the flags/args relevant to that subcommand.

### Step 3 (optional): Nest deeper for `help`

The `help` subcommand itself accepts sub-subcommands (like `rgit help add`), so it gets its own `->` dispatch:

```zsh
(help)
    _arguments "${_arguments_options[@]}" : \
        ":: :_rgit__subcmd__help_commands" \
        "*::: :->help" \
    && ret=0

    case $state in
    (help)
        words=($line[1] "${words[@]}")
        (( CURRENT += 1 ))
        curcontext="${curcontext%:*:*}:rgit-help-command-$line[1]:"
        case $line[1] in
            (add)
                _arguments "${_arguments_options[@]}" : && ret=0
                ;;
            ...
        esac
        ;;
    esac
    ;;
```

## 7. `_arguments` Options: `-s -S -C`

```zsh
if is-at-least 5.2; then
    _arguments_options=(-s -S -C)
else
    _arguments_options=(-s -C)
fi
```

- `-s` — allow option stacking. With this, `rgit -hV` is equivalent to `rgit -h -V` (single dash, multiple short flags).
- `-S` — "don't stop at the first non-option argument". With `-S`, after `rgit add -h`, the `-h` is still treated as an option, not a positional arg.
- `-C` — adds the `-C` option to `_arguments` (context-based completion caching).

`is-at-least` checks the Zsh version. `-S` was added in Zsh 5.2, hence the conditional.

---

## 8. `&& ret=0`

```zsh
_arguments ... && ret=0
```

- `_arguments` returns 0 on success, non-zero on failure.
- `ret` starts at 1 (line 8: `local ret=1`). If `_arguments` succeeds, `ret` becomes 0.
- The completion function's exit code tells Zsh whether completion succeeded. 0 = success.

---

## 9. Lazy Registration Pattern (Bottom of File)

```zsh
if [ "$funcstack[1]" = "_rgit" ]; then
    _rgit "$@"
else
    compdef _rgit rgit
fi
```

This is a dual-purpose trick:

- **When sourced** (e.g., `source _rgit` in `.zshrc`): `$funcstack[1]` is the source file or `zsh`, not `_rgit`. So it runs `compdef _rgit rgit`, which registers `_rgit` as the completer for the `rgit` command.

- **When called by Zsh** (for actual completion): `$funcstack[1]` is `_rgit`. So it runs `_rgit "$@"` and performs the actual completion logic.

This means the same file can be both sourced for registration AND executed for completion, without needing a separate registration step.

---

## 10. Summary: The Flow

When you type `rgit <TAB>`:

1. Zsh calls `_rgit`
2. The top-level `_arguments` specs are evaluated — it offers `-h`, `--help`, `-V`, `--version` as completions, plus the output of `_rgit_commands` (all subcommand names like `add`, `commit`, etc.)
3. User types `rgit add <TAB>`
4. `->rgit` triggers, `$state` becomes `rgit`
5. The `case $state` block runs, dispatches to the `add` case
6. A new `_arguments` call offers `-h`, `--help`, and `paths` with `_files` completions

---

## Quick Reference

| Pattern | Meaning |
|---|---|
| `#compdef cmd` | This file completes `cmd` |
| `'-f[desc]'` | Short flag `-f` |
| `'--flag[desc]'` | Long flag `--flag` |
| `'-f+[desc]:pl:action'` | Flag taking a value |
| `':label:action'` | Required positional arg |
| `'::label:action'` | Optional positional arg |
| `'*::label:action'` | Repeatable (greedy) arg |
| `':->tagname'` | State dispatch (sets `$state`) |
| `":: :funcname"` | Delegate to another function |
| `_files` | Complete file paths |
| `_default` | Generic fallback completion |
| `_describe -t tag 'heading' arr` | Show array as completions |
| `(($+functions[name])) \|\| name(){...}` | Define function if not exists |
| `_arguments ... && ret=0` | Mark success |
